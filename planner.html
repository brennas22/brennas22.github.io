The main design problem I worked on was the Case Detail Form, which consisted for a number of sub-problems:
+ Concurrent cases
+ Main form structure
+ Error states
+ Preop workup / cards
+ Return to surgery

Problem

LiveData was looking to redesign a clunky and inefficient form used to collect patient information, schedule procedures, mark preop workup items, assign surgeons and more. Some of the main problems that needed to be addressed were:

+ Organization of information
- How do people filling out the form quickly know what section information goes in?
- Can people scan the form and understand it?
- Is there a way to only complete the required sections?

+ Keyboard Navigation / 508 Compliance
- Can the entire form be navigated in an intuitive way via keyboard only?
- Can a user who is colorblind, low vision, etc. understanding and interact with the form?

+ Error States
- How do users know that an error has been made?
- How do users know if an error has been corrected?

+ Skills Utilized:
- Adobe XD
- User Testing
- Principle
- Design Research
- Wireframing

Process Book

Below is a closer look at my process for reaching the final products above.

Cards Metaphor

The first project I worked on was a metaphor for "Lists UI," which was necessary in multiple places throughout the Case Detail Form.

1. Preop Workup items need to be distinguished from one another, need to be easily added/removed, need to be easily checked/unchecked
2. Other lists such as pharmacy items, special supplies etc. have additional secondary ifnromation that needs to be displayed. They also need to be easily distinguished, easily added/removed, and easily have their quantity adjusted

This was an ongoing design problem, which started before the implementation the entirely new form structure.

Iteration 1:

This iteration occurred before the redesign of the form

My first design attempt was built into the existing form at the time. There needed to be a way to input different pieces of data for one item (it later turned out this was all linked in the backend).

My initial thought was to just enter each piece of data, one at a time, and they would appear as a line in the form.

This design was flawed though because there was no way of knowing what the next information you needed to know would be, and what if you didn't know a portion? How would you leave it blank and go to the next section? It is also very unclear which element has focus.

Iteration 2:

This iteration occurred before the redesign of the form

My second design included an add button so that keyboard nav wouldn't have to tab through all the sections if they weren't going to add anything. It also had all the sections visible so that some could be skipped if desired and so that all information was visible initially. Additionally, I added the ability to edit, not just remove, entries.

In the video here I also included the hierarchy that would be followed by keyboard navigation (second half of video)

This design was flawed also because of its clunkiness - in particular the fact that keyboard nav went up to reach edit, which doesn't follow the flow of the rest of the form. It also took far to many clicks / tab presses to reach the save button.

Iteration 3:

This iteration occurred before the redesign of the form

My third design mimicked the existing metaphor that was being used for pre-op workup items (a similar style of items), which consisted of a popup modal dialog where the information is input. I set it up as a grid to most easily accomodate the different sections.

The grid set up also allows for simple keyboard navigation and, as in iteration 2, the add button negates the need to tab through the whole thing if it's not necessary to access it.

Iteration 4 (Final Iteration):

Between iteration 3 and 4 it was determined that the entire form should be redesigned for efficency and ease of use. This iteration occurred during that redesign.

Once the form was streamlined, it became clear that a card metaphor would be the best option. (Also at this point we had learned that we didn't need separate input fields, instead inputs that differ only slightly are still listed as completely separate options in the dropdown menu).

The card has many benefits in this case: easily keyboard navigable (and easy to skip over), cards can highlight key title information, but also provide a cohesive option for items with more details. They can be easily altered in terms of quantity, checkboxes, and removal.

Research

When we decided to redesign the entire form, the next thing I did was begin researching best practices for forms, which I did primarily via medium and other design resources.

Designing More Efficient Forms: Structure, Inputs, Labels and Actions
by Nick Babich - read more

Key Takeaways:
+ Group related information
+ Forms should never consist of more than one column
+ Clearly distinguish required and not required fields
+ Sentence case is more readable than Title Case (and never use ALL CAPS)
+ Top labels are better than side labels
+ If using inline labels (placeholder text), move label to top when user begins typing

Designing More Efficient Forms: Assistance and Validation
by Nick Babich - read more

Key Takeaways:
+ Make sure to have inline validation (inform the user right away)
+ User color to indicate success/failure
+ Provide a clear error message

Form Design

The first iteration focused on the overall structure of the interaction the user has with the form. The first thing I focused on was labels - specifically how to handle inline versus top navigation, and switching the form to be single column only.

Animation

I used the software Principle to mock up the animations. Initially, there is an inline label in each textbox, but when focus is directed there the label pop up to the top and shows focus in blue. When a textbox has been filled but is not in focus the label appears on top, in black.

I also did the animation for the next button (although this is not it's final state). The animation shows the transition from unfocused/hover state, to focused, to clicked.

Error Handling

The next problem I tackled was error handling, considering inline validation, final valiation, and error correction

The error handling I designed uses both color and other means to effectively communicate errors. 508 accesibility compliance standards require that color not be the only means of indication. A subtle alert indicates an error as soon as it's been made, allowing the user to rapidly correct before they've made multiple errors that can become overwhleming.

If a user tries to move on to another section before correcting errors they will recieve a stronger indication at the top with a specific diagnosis of the error, as well as an indicator in the lefthand menu.

Something that had been discovered in the course of LiveData's development was that some users click save at the very beginning in order to see what fields are required and then only fill those in. The third image indicates this view - all sections with errors are indicated on the left and errors can be navigated one by one with the arrows at the top.

The same arrow metaphor applies when users are fixing errors. When an error is remedied the status bar turns green, provides a fix message, and the arrows can be used to navigate to the next error that requires fixing.

Concurrent Cases

Concurrent cases were difficult to display and navigate because between cases there is some information that overlaps and a lot that doesn't. The main challenge was clearly and efficiently switching between cases.

Iteration 1:

My first design attempted to continue the metaphor of the bubbles on the left by giving sub bubbles for each individual case, with an overhead "concurrent" bubble that covers the data consistent across the two cases. However, this design takes up too much space on the left side and also does not appear visually coherent.

Iteration 2:

My second iteration attempted to simplify the design by switching to only "indvidual" and "all cases" and then navigating between the cases at the top. However, this design was awkward to use, especially when considering keyboard navigation because a user would need to change things in both the x and y directions.

Iteration 3:

Because of the confusion of making changes in both the x and y directions, I moved everything to the top, giving the "all cases" hierarchy above the individual cases. However, this takes up an uneccessary amount of vertical space, and the distinction between all cases and the invidual cases is already clear enough from the text.

Iteration 4 (Final Iteration):

The iteration that we ended up selecting turned out to be the most simple. The text provides enough distinction between the 3 sections so that they can remain on one line, with a save mechanism between sections. Each case has the individual sections to be completed and the all cases consists of just one page, with the rest greyed out.

Return to Surgery

My next task was to redesign the specific module for return to surgery. This is used to indicate whether the procedure being currently planned is the result of a previous surgery (of which there could be multiple) and also to indicate whether this return is planned (i.e. second hip replacement) or unplanned (i.e. complications), which the previous version could not do.

User Feedback

I had the opportunity to speak with a "working group" of surgeons, OR schedulers, circulating nurses etc. to conduct user testing and recieve feedback on designs.

Key Takeaways

This was the first project I worked on at LiveData. I learned a lot about form structure and navigation efficiency as well as accessibility requirements such as color, contrast, size, and keyboard navigation. I also learned a lot about human behavior (aka users never do what you want them to) and how important visual cues are.
